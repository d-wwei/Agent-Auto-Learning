# Auto-Learning 大白话指南

## 一句话说清楚

**装了这个东西之后，AI Agent 会变成一个"有记性的助手"。你教过它的东西它会记住，它踩过的坑下次不会再踩，它搞定过的复杂事情会变成一套"操作手册"留着以后用。**

没装之前，Agent 每次对话都是从零开始的，跟失忆了一样。装了之后，它会越用越好用。

---

## 具体是怎么发挥作用的？

### 场景 1：你纠正了 Agent 一次，它永远记住

**没装之前：**

```
第 1 天：
你：帮我装个依赖
Agent：好的，运行 npm install xxx
你：我们项目用的是 pnpm，不是 npm！
Agent：抱歉，pnpm install xxx

第 2 天（新对话）：
你：再装一个依赖
Agent：好的，运行 npm install yyy      ← 又忘了！
你：……我昨天不是说了用 pnpm 吗
```

**装了之后：**

```
第 1 天：
你：帮我装个依赖
Agent：好的，运行 npm install xxx
你：我们项目用的是 pnpm，不是 npm！
Agent：抱歉！
      （内部动作：调用 learn → 保存一条记忆："用户项目用 pnpm，不用 npm"）

第 2 天（新对话）：
你：再装一个依赖
Agent：（内部动作：调用 recall → 搜到"用户项目用 pnpm"）
Agent：好的，运行 pnpm install yyy     ← 记住了！
```

**效果：你纠正一次，Agent 永远记住。不管隔多久、开多少个新对话，这条纠正都在。**

---

### 场景 2：Agent 自己踩过坑，下次直接绕过

**没装之前：**

```
第 1 次：
你：帮我配置 ESLint
Agent：好的，我来创建 .eslintrc.json...
      （折腾 10 分钟，发现 ESLint 9 不支持这个格式了）
      改用 eslint.config.js...
      （又折腾 5 分钟，搞定了）
Agent：配好了！

第 2 次（新项目，新对话）：
你：帮我配置 ESLint
Agent：好的，我来创建 .eslintrc.json...    ← 又走一遍弯路！
      （又折腾 10 分钟...）
```

**装了之后：**

```
第 1 次：
你：帮我配置 ESLint
Agent：（先调用 recall → 没找到相关知识）
      好的，我来创建 .eslintrc.json...
      （折腾发现不行，改用 eslint.config.js，搞定了）
Agent：配好了！这次踩了个坑，我来记一下。
      （调用 learn → review 引擎自动提取：
        记忆1："ESLint 9 不支持 .eslintrc.json，必须用 flat config"
        记忆2："typescript-eslint v8 有个 tseslint.config() 很好用"）

第 2 次（新项目，新对话）：
你：帮我配置 ESLint
Agent：（先调用 recall → 找到2条相关记忆）
      我之前学到过：ESLint 9 要用 flat config，用 tseslint.config() 最方便。
      （直接创建 eslint.config.js，一次搞定）    ← 没走弯路！
```

**效果：Agent 第一次踩坑花了 15 分钟，第二次 2 分钟搞定。知识积累让同类任务越来越快。**

---

### 场景 3：复杂操作变成"操作手册"

**没装之前：**

```
你：帮我把服务部署到 staging
Agent：（查日志 → 试了3种方法 → 发现要先构建 Docker 镜像 →
       发现架构要指定 amd64 → 推到 ECR → 更新 ECS 任务定义 →
       等健康检查通过 → 终于成功了）
Agent：搞定了！

一个月后：
你：再帮我部署一下
Agent：（又从头摸索...又花了30分钟）
```

**装了之后：**

```
你：帮我把服务部署到 staging
Agent：（经过同样的试错过程，搞定了）
      这个流程挺复杂的，我把它保存成一个技能（skill）吧。
      （调用 learn → 创建技能 "deploy-to-staging"，记录了完整步骤）

一个月后：
你：再帮我部署一下
Agent：（调用 recall → 找到 deploy-to-staging 技能 → 自动加载完整步骤）
      我有这个操作的完整流程！按步骤来：
      1. 构建 Docker 镜像（--platform linux/amd64）
      2. 推送到 ECR
      3. 更新 ECS 任务定义
      ...
      （5分钟搞定）    ← 从30分钟变成5分钟
```

**效果：复杂工作流不再是"每次重新发明轮子"，而是变成可复用的操作手册。**

---

### 场景 4："操作手册"会自动更新

```
你：帮我部署
Agent：（调用 recall → 加载 deploy-to-staging 技能）
      按之前的流程...等等，第 3 步用的 ECR 地址变了。
      （发现技能里的 ECR 地址过时了，立即修补）
      （调用 learn → 更新技能中的 ECR 地址）

下次部署时，技能已经是最新的了。
```

**效果：技能不是写死的，Agent 在使用过程中发现问题会自动修正。知识是活的。**

---

### 场景 5：记住你的沟通偏好

```
你：（第一次交互）帮我总结一下这个会议
Agent：（写了一大段，最后还加了"综上所述..."的总结段落）
你：太啰嗦了，以后结论优先，不要最后的总结段

Agent：（调用 learn → 保存偏好："结论优先，简洁直接，不要结尾总结"）

从此以后，所有回复都是结论优先、简洁直接的风格。
不管是写代码、写文档还是回答问题，这个偏好都会生效。
```

**效果：Agent 适应你的风格，而不是你每次都要适应 Agent。**

---

### 场景 6：你什么都不做，它自己在学

这是最关键的场景 — **你不需要主动说"帮我记住xxx"**。

装了认知底座之后，Agent 的行为模式发生了根本变化：

| 以前 | 现在 |
|------|------|
| 接到任务就开始干 | 先搜索一下有没有之前学到的相关知识 |
| 被纠正了说"好的" | 被纠正了立刻保存这个纠正 |
| 发现了一个坑口头说"注意" | 发现了一个坑自动保存为记忆 |
| 完成任务就结束 | 复杂任务完成后自动复盘，提取值得保留的经验 |

**你正常用 Agent，它在后台默默变聪明。**

## 总结：装了之后有什么效果？

| 效果 | 说明 |
|------|------|
| **不再重复犯错** | 你纠正过的，它永远记住 |
| **不再重复提问** | 你说过的偏好，它跨会话保持 |
| **同类任务越来越快** | 踩过的坑变成知识，下次直接绕过 |
| **复杂流程可复用** | 试错得来的多步操作变成"操作手册" |
| **知识会自动更新** | 发现过时的知识会自动修正 |
| **你不需要刻意操作** | 认知底座让 Agent 自动具备学习意识 |
| **人类可以审计** | 所有学到的东西都是 Markdown 文件，随时可以看 |

## 它是怎么做到的？（极简版）

三个东西配合工作：

**1. 认知底座（~30 行规则）**
写在 Agent 的系统配置里，让 Agent "天生就想学习"。
就像你跟新员工说"遇到问题先查 wiki，学到新东西要记笔记" — 这 30 行规则就是这个作用。

**2. 三个工具**
- `recall` = "帮我想想之前学过什么"（搜索记忆 + 技能）
- `learn` = "帮我记住这个"（保存记忆 / 创建技能 / 自动复盘）
- `learning_status` = "我现在学了多少东西"（查看统计）

**3. 存储**
所有东西存在 `~/.auto-learning/` 文件夹里：
- 记忆 → 一个个 `.md` 文件，你打开就能看懂
- 技能 → 一个个 `SKILL.md` 文件，就是操作手册
- 索引 → SQLite 数据库，让搜索很快

## 一个完整的使用周期

```
Day 1 — 第一次用
├── Agent 搜索先验知识 → 什么都没有（刚装的）
├── 干活过程中踩了几个坑
├── 你纠正了 Agent 两次
├── Agent 自动保存了 3 条记忆 + 1 个技能
└── 你完全不需要管这些，正常干活就行

Day 2 — 第二次用
├── Agent 搜索先验知识 → 找到昨天的 3 条记忆
├── 不再犯昨天纠正过的错误
├── 遇到类似任务，直接用昨天学到的技能
├── 又学到了 2 条新知识
└── 总知识量：5 条记忆 + 1 个技能

Day 30 — 用了一个月
├── Agent 已积累 50+ 条记忆，5 个技能
├── 它知道你喜欢什么风格、项目用什么技术栈
├── 常见任务都有对应的技能，效率很高
├── 过期或低质量的记忆被自动清理
└── 你可以随时 npm run dashboard 看它学了什么

                    就像一个越来越懂你的助手
```
